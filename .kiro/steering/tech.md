# T-SQL Remaker - Technical Context

## 技術スタック

### プログラミング言語
- **Rust** 2021 edition
  - 高性能かつメモリ安全
  - 強力な型システム
  - 優れたエラーハンドリング

### 主要ライブラリ
- `thiserror`: エラー型の定義
- `once_cell`: 静的データの初期化
- `proc-macro2`, `quote`, `syn`: マクロ処理（将来予定）

### 開発ツール
- **Cargo**: 依存管理とビルド
- **rustfmt**: コードフォーマット（100文字制限）
- **clippy**: Lint（警告はエラーとして扱う）
- **cargo-tarpaulin**: カバレッジレポート（80%以上）

## アーキテクチャ原則

### 単一責任の原則
各クレートは単一の責務を持つ

### 依存逆転の原則
上位モジュールは下位モジュールに依存せず、抽象（AST）に依存する

### 開放閉鎖の原則
拡張には開いており、修正には閉じている

## 非機能要件

### パフォーマンス
- 1000行のストアドプロシージャを1秒以内に処理

### 信頼性
- パニックを起こさない（ライブラリコード）
- 全てのエラーが `Result` 型で返される

### 保守性
- 各方言の Emitter は独立して開発可能
- 新しい AST ノードの追加が既存コードに影響しない

## テスト戦略

- **単体テスト**: 各モジュールの独立テスト
- **統合テスト**: クロスコンポーネントのテスト
- **プロパティベーステスト**: 有効な入力に対する挙動検証
- **カバレッジ**: 80%以上、クリティカルパスは90%以上

## 設計プロセスの基準（design.md / spec-tasks）

> レトロスペクティブ 2026-07-09 で特定: 設計書の過少指定（構造体フィールド形状の省略、未定義型の dangling 参照）が実装者への推測転嫁とエミッタ期待との乖離を生む。

全ての design.md / spec-tasks は以下を満たす:

1. **構造体フィールドの完全列挙**: enum バリアントだけでなく、各構造体の全フィールド（名前・型・可視性）を明示。
2. **参照型の完全定義**: 設計書が参照する型は全て定義済み（dangling 参照禁止）。未定義型があれば `validate-design` / `kiro:spec-tasks` gate を fail させる。
3. **暗黙契約の文書化**: Visitor の no-auto-descent、精度に依存する値の表現（f64 vs String）等を明示。
4. **完了宣言は nextest 裏付け**: 「実装済み」「check clean」「N%完了」の主張は `cargo nextest run --workspace` の貼り付け出力で裏付ける。lib-only `cargo check` は不可（テストコンパイル崩壊を隠す）。
