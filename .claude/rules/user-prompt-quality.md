# User Prompt Quality Guide

ユーザーがAI（Claude Code）に機能開発を依頼する際のプロンプト品質ガイドライン。
「適当な考慮」による手戻りコストを最小化する。

## 原則

**プロンプトの品質が実装の品質を決める**

AI は明確な要件・ユースケース・受入基準が与えられた場合に最も高い成果を出す。
逆に、曖昧な指示は実装の手戻りと品質のばらつきを生む。

---

## 必須要素

### 1. ペルソナ（誰が使うか）

**❌ 悪い例:**
```
Language Serverを設計して開発してほしい
```

**✅ 良い例:**
```
SAP ASE開発者（500行以上のストアドプロシージャを日常的に書く）が
VSCode/ZedでASE開発を行うためのLanguage Serverを開発したい
```

### 2. 動機（なぜ必要か）

**❌ 悪い例:**
```
Hover機能を実装して
```

**✅ 良い例:**
```
ASE開発者は長大なプロシージャ内で変数の定義元を頻繁に確認する。
Hover機能により、マウスオーバーだけで型情報やスキーマ情報を表示したい
```

### 3. ユースケース（入力→期待出力）

**❌ 悪い例:**
```
Rename機能を実装して
```

**✅ 良い例:**
```
Rename機能:

UC-1: ローカル変数のリネーム
入力: DECLARE @count INT; SET @count = 1; SELECT @count
カーソル位置: @count 上
new_name: @total
期待出力: 3箇所のTextEdits（@count → @total）

UC-2: テーブル名のリネーム（大文字小文字混在）
入力: CREATE TABLE Users ...; SELECT * FROM users; INSERT INTO USERS ...
カーソル位置: Users 上
期待出力: 3箇所以上のTextEdits（Users/users/USERS → 全て新名称）

UC-3: エッジケース - @なしの変数名は拒否
入力: DECLARE @count INT; カーソル @count 上、new_name = "total"（@なし）
期待出力: None（リジェクト）
```

### 4. 受入基準（検証可能な条件）

**❌ 悪い例:**
```
テストを書いてください
```

**✅ 良い例:**
```
受入基準:
- [ ] 正常系3パターンのテストがパスすること
- [ ] エッジケース2パターンのテストがパスすること
- [ ] パース失敗SQLでgracefulにフォールバックすること
- [ ] cargo clippy -- -D warnings がパスすること
- [ ] 890+テストが全て通ること
```

### 5. 非スコープ（やらないこと）

**❌ 悪い例:**
```
（省略 — やらないことが未定義）
```

**✅ 良い例:**
```
非スコープ:
- 複数ファイルにまたがるリネーム（現在は単一ファイルのみ）
- テーブル名がスキーマ名を含む場合 (dbo.users)
- CREATE TRIGGER 構文（Parser未対応）
```

## 過去の失敗パターンと教訓

| 失敗 | 原因 | テンプレートで防げるか |
|------|------|---------------------|
| Hover で何を表示するか不明 | 表示内容が未定義 | ユースケースで期待出力を明記 |
| parse_with_errors の誤解 | APIの挙動を推測 | 制約にAPI仕様を明記 |
| CREATE UNIQUE INDEX でクラッシュ | Parser未対応を考慮せず | 非スコープに明記 |
| lsp-types 0.94 と 0.97 の混乱 | バージョン固有APIを確認せず | 制約にバージョン情報を明記 |
| モノリシックコミット | コミット粒度が未定義 | 受入基準に粒度要件を含める |

---

## チェックリスト

プロンプトを送信する前に:

- [ ] ペルソナを定義した（誰が、どんな状況で使うか）
- [ ] 動機を説明した（なぜ必要なのか）
- [ ] ユースケースを具体的に書いた（入力→期待出力）
- [ ] エッジケースを含めた
- [ ] 非スコープを明記した（やらないこと）
- [ ] 受入基準が検証可能である（テストで確認できる）
- [ ] 制約を明記した（技術的制約、バージョン等）
