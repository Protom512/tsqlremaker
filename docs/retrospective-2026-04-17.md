## PDCA サイクル分析

### Cycle 1: Lexer/Parser 基盤構築

```
Plan:  tsql-lexer と tsql-parser の設計・実装
Do:    スペック駆動で実装（.kiro/specs/sap-ase-lexer, tsql-parser）
Check: テスト通過、カバレッジ確認
Act:   並列実装ワークフロー(.claude/rules/parallel-impl-workflow.md)を確立
```

**評価**: Plan と Do は良かった。スペック駆動が効果的。
**改善点**: Check でカバレッジの定量測定が不十分。

### Cycle 2: LSP Phase 1-3 実装

```
Plan:  「DB LS デファクトスタンダード調査」→「実装」の順
Do:    Phase 1-3 を一括実装（462d12a の巨大コミット）

**評価**: Do が粗い。Check は通ったが品質の深い評価が不在。Act が欠落。
**改善点**: フェーズごとに独立したコミット＋レビューが必要。

### Cycle 3: LSP Phase 4 実装

```

## ユーザー側の改善ポイント

### プロンプト品質の課題

| 課題 | 具体例 | 改善案 |
|------|--------|--------|
| 要件が曖昧 | 「Language Serverを設計して」→ スコープ不明 | 「Phase N の機能Xを実装する。入力→出力のシナリオは以下の通り...」 |
| ユースケース不在 | 「Hoverを実装」→ 何を表示するか未定義 | 「CREATE TABLE名にホバー→カラム一覧表示。DECLARE変数→型情報表示」 |
| 受入基準なし | 「テストを書いて」→ 何をテストすべきか不明 | 「正常系3パターン、エッジケース2パターンのテストがパスすること」 |
| 設計意図の欠落 | なぜその機能が必要かが不明 | 「ASE開発者は500行以上のストアドプロシージャを書くため、ナビゲーション機能が必須」 |

### 推奨プロンプトテンプレート

```
## 機能要求: [機能名]

### 背景

### ユースケース

2. [シナリオ2]: 入力 → 期待出力

### 受入基準
- [ ] [具体的な検証可能な条件1]
- [ ] [具体的な検証可能な条件2]

### 制約
- [技術的制約、バージョン互換性等]
```
