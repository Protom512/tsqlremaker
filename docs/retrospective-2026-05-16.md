## KPT 分析

### Keep（継続すべきこと）

#### K1: ハーネス（.claude/）の段階的進化

**事実**: セッションごとにルール・スキルを追加し、反復的に改善した。
- Phase 1-3の教訓から `dependency-version-compatibility.md`, `project-ast-types.md` を作成
- レビュー対応の教訓から `lsp-use-case-template.md`, `pre-implementation-checklist.md` を作成
- ブランチ管理の混乱から `git-branch-strategy.md` を作成

**なぜ良かったか**: 各セッションで踏んだ罠がルール化され、後のセッションで同じ轍を踏まなくなった。
例: `parse_with_errors()` の誤解 → `pre-implementation-checklist.md` に「API署名をソースで確認」が追加され、
以降のAPI罠（RenameParams フィールド名、symbol() 戻り値）を事前に防止できた。

#### K2: テストファースト・ブラックボックステスト

**事実**: 全モジュールで公開APIのみをテストし、プライベート実装への結合を避けた。
テスト数: code_actions(6) + rename(8) + workspace_symbols(9) + references(8) + hover(10) + completion(15) + ...

**なぜ良かったか**: リファクタリング時にテストが壊れなかった。
`token_matches_symbol()` の抽出時も、rename/references のテストは一切修正不要だった。

#### K3: 認知負債フレームワークの適用

**事実**: 「理解負債→認知負債→意図負債」の分類で問題を構造化し、優先度付けした。
- 理解負債（db_docs.rs の3箇所データ重複）→ P0 で即解消
- 認知負債（is_definition_token 不十分）→ P1 で解消
- 意図負債（設計判断が未記録）→ ADR文書化で解消

**なぜ良かったか**: 「何を直すべきか」の判断が明確になり、限られた時間で最大の効果を得られた。

#### K4: 線形PRチェーンと並列実行のハイブリッド

**事実**: 依存関係のある変更は線形チェーン（#38→#40→#39→#41...）で、
独立した変更（P2-2/P2-3/P2-4）は3エージェント並列で実行。

**なぜ良かったか**: 各PRのdiffが100行以下に収まり、レビュー可能性が確保された。
並列実行は P2 の3タスクを7分で完了させた。

#### K5: MEMORY.md によるセッション間コンテキスト維持

**事実**: プロジェクトメモリに重要なAPI事実（lsp-types 0.94 の罠、Parser未対応構文、SymbolTable仕様）を蓄積。

**なぜ良かったか**: セッションをまたいで「前回どこまでやったか」「何に注意すべきか」を即座に想起できた。
新規セッションでの立ち上がり時間が大幅に短縮された。

#### K6: DocEntry構造化データへの移行

**事実**: `db_docs.rs` で hover/signature_help/completion の3箇所に散在していたデータを統合。
syntax文字列のパース → `params: Vec<ParamInfo>` 構造体への移行。

**なぜ良かったか**: 「括弧混入」バグの根本解決。データが間違っていたらデータを直す、パーサーで頑張らない原則。

#### K7: ADR（Architecture Decision Record）の文書化

**事実**: 3つのADRで「なぜそう決めたか」を明文化（symbol table基盤設計、db_docs統合、completion戦略）。

**なぜ良かったか**: 将来の開発者（とAI）が設計意図を理解でき、不用意な変更を防げる。

### Problem（課題）

#### P1: 要件定義・ユースケース定義の不足（ユーザー側課題）

**事実**: ユーザー自身が「要件定義、設計、ユースケースなどが適当な考慮だった」と自己評価。
具体的な影響:
- 「Hoverを実装する」だけで、CREATE TABLE名にホバーしたら何を表示するかが未定義
- ペルソナ（誰が使うか）が不明確だったため、機能の優先順位付けが場当たり的
- 受入基準が「テストが存在する」程度で、シナリオレベルの具体性が不足

**影響**: 実装と期待のギャップが生じ、レビューでの手戻りが発生。
Hoverの表示内容、Rename の検証ルール（@プレフィクス必須）などを実装中に決定。

**根因**: スペック駆動開発（.kiro/specs/）のプロセスが Lexer/Parser には適用されたが、
LSP機能には適用されなかった。AIへの指示が「Language Serverを設計して開発して」と曖昧だった。

#### P2: モノリシックコミット（Phase 1-3）

**事実**: Phase 1-3（462d12a）が462行の全機能を1コミットに詰め込み、レビュー困難。

**影響**: ロールバック粒度が粗く、どの変更がどの機能に対応するか不明。

**根因**: フィーチャーブランチ戦略が事前に明文化されていなかった。
Phase 4では改善（1機能=1コミット）したが、Phase 1-3の負債は残った。

#### P3: 実装前のAPI確認不足

**事実**:
- `parse_with_errors()` → 「部分結果を返す」と想定 → 実際は Err を返す
- `RenameParams.text_document_position` → 他バージョンの `text_document_position_params` と混同
- `tower-lsp symbol()` → `WorkspaceSymbolResponse` と想定 → 実際は `Vec<SymbolInformation>`
- `DocumentSymbol.deprecated` → `#[deprecated]` に気づかずコンパイルエラー

**影響**: 各API罠の発見と修正に30分〜1時間を消費。合計で4-5時間の無駄。

**根因**: 「推測でAPIを使う」習慣。実装前にソースコードで署名を確認する習慣がなかった。
→ `pre-implementation-checklist.md` で対策済みだが、初期段階では未整備だった。

#### P4: Parser能力の事前把握不足

**事実**: `CREATE UNIQUE INDEX`, `ALTER TABLE`, `CREATE TRIGGER`, `EXEC` が未対応。
実装中に発見し、都度回避策を講じた。

**影響**: Code Actions の INSERT skeleton 生成で `parse_with_errors()` が失敗 →
`build_resilient_symbol_table()` のフォールバック機構が必要になった。

**根因**: Parser の対応範囲を事前に調査しなかった。
→ `project-ast-types.md` に「Parser未対応構文」セクションを追加して対策済み。

#### P5: LSP統合テストの欠如

**事実**: `server.rs`（393行）にテストが0件。LSPプロトコルレベルのE2Eテストが存在しない。

**影響**: ハンドラーのルーティング、パラメータ渡し、エラーレスポンスが未検証。
ユニットテストは各モジュールにあるが、結合ポイントの欠陥を検出できない。

**根因**: テスト戦略がユニットテストのみに偏っていた。
tower-lsp のテストユーティリティ（`TestService`）の存在を後で知った。

#### P6: Ralphセッションの断絶

**事実**: Ralphループが4回以上中断・再起動。各再起動で状態の再検証が必要だった。

**影響**: セッション間のコンテキスト復元に時間を消費。
前セッションで既に修正済みのエラーに気づくのに時間を要した。

**根因**: コンテキストウィンドウの制約と、セッション間でのタスク状態の永続化不足。

#### P7: cargo fmt忘れによるCI失敗

**事実**: レビュー対応後のコミットで fmt を忘れ、PR #39 のCI Lint が FAILURE。

**影響**: CIの再実行待ちとコミットの追加が必要に。

**根因**: エージェントプロンプトに `cargo fmt` を含めていても実行漏れが起きる。
人間の確認に依存する品質ゲートは不十分。

#### P8: 線形PRチェーンのコンフリストリスク

**事実**: `#38 ← #40 ← #39 ← #41,#42,#43,#44` の依存関係で、
PR #43 が #44 のコミットを巻込み、rebase で解消に手間。

**影響**: コンフリクト解消に30分以上消費。

**根因**: 依存関係のあるPRを線形に積んだため、ベース変更が下流に波及。
独立してマージできる粒度にすべきだった。

### Problem（課題）

#### P2-1: 移行戦略の順序エラー — deprecatedマーク先行の罠

**事実**: Phase 0-Aで以下の順序で作業した:
1. LineIndex を新規実装 ✅
2. 旧関数に `#[deprecated]` を付与 ⚠️
3. 旧関数の呼び出し元の移行 ❌（未着手）

結果: 50件の `use of deprecated function` clippyエラーが発生し、
`-D warnings` でビルドが失敗。

**影響**: Phase 0-Aが完了せず、ブランチに未コミットの変更が残った。

**根因**: 「新APIを作って旧APIをdeprecatedにする」戦略は、
呼び出し元の移行と同時に実施しなければならないことを理解していなかった。
正しい順序は: **新API実装 → 呼び出し元全移行 → deprecatedマーク**。

#### P2-2: 一度に多すぎる作業の着手

**事実**: このセッションで以下を同時に進行:

**影響**: Phase 0-Aの完了（本来の目標）に集中できず、
前半のIssue登録で大部分のコンテキストを消費。

**根因**: 「分析→計画→実行」の全てを1セッションで行おうとした。
分析と計画は別セッションで完了すべきだった。

#### P2-3: WIPコミットの欠如

**事実**: Phase 0-Aの変更が全てワーキングツリーに未コミット:
- `line_index.rs` (新規237行, untracked)

**影響**: セッション断絶時に全ての変更が失われるリスク。
コンテキストコンパクション後の再開コストが高い。

**根因**: 「テストが通ってからコミットする」という原則への過度な固執。
WIPコミット（`--no-verify`）は進行中の作業を保護する有効な手段。
